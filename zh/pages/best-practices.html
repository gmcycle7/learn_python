1. 命名慣例

好的名稱讓程式碼具有自說明性。Python 有在 PEP 8 中定義的強烈慣例。

項目慣例範例
變數snake_casetotal_price、user_name
函式snake_casecalculate_area()、get_user_input()
類別PascalCaseDataProcessor、HttpClient
常數UPPER_SNAKE_CASEMAX_RETRIES、PI
模組/檔案snake_casedata_utils.py、file_handler.py

好的名稱 vs 不好的名稱

# 不好 — 模糊、單字母或誤導的名稱
x = 3.14
l = [1, 2, 3]          # 'l' 在很多字型中看起來像 '1'
def calc(a, b):
    return a * b

# 好 — 清楚、描述性、揭示意圖的名稱
pi = 3.14
scores = [1, 2, 3]
def calculate_rectangle_area(width, height):
    return width * height
何時可以用單字母

迴圈計數器（i、j、k）、 座標（x、y、z）以及 短數學公式中變數含義從上下文顯而易見的情況 是可以接受的。其他所有地方，請使用描述性名稱。

MATLAB 比較

MATLAB 的文化傾向使用短變數名稱（A、 b、idx）。Python 的文化強烈偏好 較長的描述性名稱。temperature_readings 比 T 或 temps 更好。

2. 程式碼結構

組織良好的 Python 檔案遵循可預測的佈局。這使 任何人（包括未來的你）都能快速找到東西。 

#!/usr/bin/env python3
"""模組文件字串 — 一行描述此檔案的功能。"""

# === 標準函式庫匯入 ===
import os
import sys
from pathlib import Path

# === 第三方函式庫匯入 ===
import numpy as np
import pandas as pd

# === 本地匯入 ===
from my_package import helpers

# === 常數 ===
MAX_RETRIES = 3
DEFAULT_TIMEOUT = 30

# === 函式 / 類別 ===
def process_data(filepath):
    """從 filepath 讀取資料並回傳處理結果。"""
    with open(filepath) as f:
        data = f.read()
    return data.strip()

def main():
    """腳本的進入點。"""
    result = process_data("data.txt")
    print(result)

# === 主程式守衛 ===
if __name__ == "__main__":
    main()

__name__ 守衛說明

if __name__ == "__main__": 區塊只在你 直接執行檔案時運行（python my_script.py）。當檔案被作為模組匯入時 不會運行。這讓你可以重用檔案中的函式 而不會觸發主程式碼。 

# 沒有守衛 — 匯入時也會執行！
result = process_data("data.txt")   # 匯入時就執行

# 有守衛 — 只在直接執行時才運行
if __name__ == "__main__":
    result = process_data("data.txt")   # 安全
MATLAB 比較

在 MATLAB 中，每個 .m 檔案不是腳本就是函式。 Python 檔案可以兩者兼具 — __name__ 守衛是 你控制取得哪種行為的方式。

3. 註解與文件字串

註解解釋為什麼，而不是做什麼。程式碼 本身應該足夠可讀，能展示它在做什麼。

不好的註解（陳述顯而易見的事）

x = x + 1  # 將 x 加 1

# 迴圈走訪串列
for item in my_list:
    print(item)  # 印出項目

好的註解（解釋為什麼）

# 加 1 是因為感測器回報 0 基礎的通道號碼，
# 但顯示器期望 1 基礎的編號
channel_display = channel_raw + 1

# 跳過第一行 — 它包含標題，不是資料
for row in rows[1:]:
    process(row)

文件字串

每個函式都應該有一個文件字串 — 函式內部 第一行的字串，描述它的功能。 

def celsius_to_fahrenheit(celsius):
    """將攝氏溫度轉換為華氏溫度。

    Args:
        celsius (float): 攝氏溫度值。

    Returns:
        float: 華氏溫度值。
    """
    return celsius * 9 / 5 + 32
為什麼文件字串很重要

Python 的內建 help() 函式會讀取文件字串。 在 REPL 中輸入 help(celsius_to_fahrenheit) 會 顯示你的文件字串。VS Code 等 IDE 也會在你 將游標懸停在函式呼叫上時顯示文件字串作為提示。

練習 — 撰寫文件字串

為這個函式添加一個適當的文件字串：

def count_words(text):
    return len(text.split())
顯示解答 
def count_words(text):
    """回傳文字字串中的單字數量。

    單字以空白字元分隔。前後的空白字元
    會被忽略。

    Args:
        text (str): 要計算單字數的輸入文字。

    Returns:
        int: 以空白分隔的單字數量。
    """
    return len(text.split())

4. 何時撰寫函式

函式是程式碼重用的基本單位。遵循 DRY 原則（Don't Repeat Yourself，不要重複自己）。

在以下情況撰寫函式：

  • 你使用相同的程式碼 2 次以上。如果你複製貼上，你應該提取一個函式。
  • 一段程式碼做一件清楚的事。給它一個描述該事的名稱。
  • 你想獨立測試一段邏輯。函式容易測試；糾纏在一起的腳本則不然。
  • 程式碼變得難以跟隨。將它拆分成命名的函式使流程更可讀。

之前：重複的程式碼

# 在三個不同地方計算面積
area1 = 3.14159 * radius1 ** 2
print(f"Area 1: {area1:.2f}")

area2 = 3.14159 * radius2 ** 2
print(f"Area 2: {area2:.2f}")

area3 = 3.14159 * radius3 ** 2
print(f"Area 3: {area3:.2f}")

之後：一個函式

import math

def circle_area(radius):
    """回傳給定半徑的圓的面積。"""
    return math.pi * radius ** 2

for r in [radius1, radius2, radius3]:
    print(f"Area: {circle_area(r):.2f}")
保持函式簡短

一個好的經驗法則：如果一個函式無法在你的螢幕上顯示完整 （約 20–30 行），它可能做了太多事。將它拆分成 更小的輔助函式。

5. 何時拆分檔案

隨著專案成長，單一 .py 檔案變得難以 瀏覽。以下是何時以及如何拆分。

應該拆分的跡象：

  • 你的檔案超過約 200–300 行。
  • 你有服務不同目的的函式群組（例如，檔案 I/O 函式 vs 數學函式 vs 顯示函式）。
  • 你想在其他專案中重用某些函式。

範例：拆分專案

# 之前：一個大檔案（main.py，400+ 行）
# 之後：組織成模組

my_project/
    main.py              # 進入點 — 呼叫其他檔案的函式
    data_loader.py       # 讀取檔案和解析資料的函式
    calculations.py      # 數學和處理函式
    display.py           # 列印和格式化函式
    constants.py         # 共用常數（MAX_SIZE、FILE_PATH 等）

如何從你自己的檔案匯入：

# main.py
from data_loader import read_csv, parse_header
from calculations import compute_average
from display import print_report
from constants import MAX_SIZE

data = read_csv("measurements.csv")
avg = compute_average(data)
print_report(avg)
什麼是套件？

套件是一個包含 __init__.py 檔案（可以是空的）的資料夾。它讓你可以將模組組織到子目錄中：from my_package.utils import helper。小型專案不需要套件 — 只要在同一個資料夾中使用分開的 .py 檔案就可以了。

6. 除錯技巧

每個程式設計師都會花時間除錯。這些技巧能幫助你 更快地找到和修正錯誤。

6.1 — Print 除錯法

最簡單且最常見的技巧。加入 print() 語句來查看每個步驟中變數的內容。 

def process_scores(scores):
    total = 0
    for i, score in enumerate(scores):
        print(f"DEBUG: i={i}, score={score}, total={total}")  # 暫時性的
        total += score
    return total / len(scores)
記得移除除錯用的 print

在分享程式碼之前，總是搜尋 "DEBUG" 或 "TODO"。在編輯器中 快速 Ctrl+F 搜尋 "DEBUG" 就能找到 遺留的 print 語句。

6.2 — 使用 assert

assert 語句驗證你的假設是否正確。 如果條件為 False，Python 會引發 AssertionError。 

def calculate_average(numbers):
    assert len(numbers) > 0, "Cannot average an empty list"
    return sum(numbers) / len(numbers)

# 這會正常運作：
calculate_average([10, 20, 30])

# 這會引發 AssertionError 並附帶你的訊息：
calculate_average([])  # AssertionError: Cannot average an empty list

6.3 — 使用 breakpoint()

Python 3.7+ 有內建的 breakpoint() 函式， 會在該行將你帶入互動式偵錯器（pdb）。 

def mystery_bug(data):
    result = []
    for item in data:
        breakpoint()  # 執行在此暫停 — 你可以檢查變數
        result.append(item * 2)
    return result

當執行暫停時，你可以輸入變數名稱來查看它們的值、 輸入 n 前往下一行、c 繼續、 q 退出偵錯器。

6.4 — 仔細閱讀錯誤訊息

Python 的錯誤訊息異常地有幫助。它們告訴你：

  • 錯誤的型別（TypeError、ValueError、NameError 等）
  • 發生錯誤的確切行號
  • 出了什麼問題的描述
先讀最後一行

最重要的資訊在錯誤訊息的底部。 從那裡開始，然後往上找到你的程式碼中（不是函式庫程式碼） 引起錯誤的那一行。

7. 如何閱讀追蹤回溯錯誤

追蹤回溯是導致錯誤的完整函式呼叫堆疊。 學會閱讀追蹤回溯是你可以培養的最有價值的 除錯技能之一。

範例：一個真實的追蹤回溯

# 程式碼
def get_average(numbers):
    return sum(numbers) / len(numbers)

def process_file(filename):
    with open(filename) as f:
        values = [int(line) for line in f]
    return get_average(values)

result = process_file("data.txt")

如果 data.txt 包含非數字的行，你會得到：

Traceback (most recent call last):           # 1. 標題
  File "main.py", line 9, in <module>        # 2. 呼叫堆疊頂端
    result = process_file("data.txt")
  File "main.py", line 6, in process_file    # 3. 堆疊更深處
    values = [int(line) for line in f]
ValueError: invalid literal for int()        # 4. 實際錯誤
  with base 10: 'hello\n'

如何閱讀（由下往上）：

  1. 第 4 行（底部） — 錯誤型別和訊息： ValueError 告訴你一個值是錯誤的。訊息說它嘗試將 'hello\n' 轉換為整數但失敗了。
  2. 第 3 行 — 錯誤發生的位置：檔案 main.py，第 6 行，在函式 process_file 內。罪魁禍首是串列推導式 [int(line) for line in f]。
  3. 第 2 行 — 呼叫它的地方：第 9 行呼叫了 process_file("data.txt")。
  4. 第 1 行 — 只是標題："Traceback (most recent call last)" — 總是一樣的。
修正方法

添加錯誤處理或資料驗證。例如，將 int(line) 包在 try/except 中，或在轉換前 過濾掉非數字的行。

常見錯誤型別速覽

錯誤型別含義常見原因
SyntaxError程式碼不是有效的 Python缺少冒號、括號不匹配、打字錯誤
NameError找不到變數/函式名稱打字錯誤、忘記定義
TypeError操作使用了錯誤的型別字串 + 整數、引數數量錯誤
ValueError型別正確，值錯誤int("hello")、無效的轉換
IndexError索引超出範圍串列只有 5 個項目卻存取 list[10]
KeyError找不到字典鍵dict["missing_key"]
FileNotFoundError檔案不存在路徑錯誤、檔名打字錯誤
AttributeError物件沒有該屬性對字串呼叫 .append()
IndentationError縮排不一致混用 tab 和空格
練習 — 閱讀追蹤回溯

這段程式碼有什麼問題，哪一行導致了錯誤？

def get_initials(names):
    initials = []
    for name in names:
        initials.append(name[0].upper())
    return initials

people = ["Alice", "Bob", "", "Diana"]
print(get_initials(people))
顯示答案

錯誤： IndexError: string index out of range 發生在 name[0].upper() 這行，當 name 是 空字串 "" 時。空字串沒有索引 0。

修正：跳過空字串：

for name in names:
    if name:  # 跳過空字串
        initials.append(name[0].upper())

8. 初學者友善風格指南（PEP 8）

PEP 8 是 Python 的官方風格指南。以下是對初學者 最重要的規則。

8.1 — 使用 4 個空格進行縮排

# 正確 — 4 個空格
def greet(name):
    print(f"Hello, {name}")

# 錯誤 — 2 個空格（在 JavaScript 中常見，但在 Python 中不是）
def greet(name):
  print(f"Hello, {name}")
永遠不要混用 tab 和空格

設定你的編輯器在按 Tab 時插入 4 個空格。混用 tab 和空格會導致 IndentationError，或更糟的是， 不可見的錯誤。在 VS Code 中，Python 檔案預設就是這樣。

8.2 — 行長度：目標 79–99 個字元

PEP 8 建議 79 個字元。許多團隊允許到 99 或 120。關鍵是： 如果你需要橫向捲動，那行就太長了。 

# 太長 — 難以閱讀
result = some_function(argument_one, argument_two, argument_three, argument_four, argument_five)

# 更好 — 拆開
result = some_function(
    argument_one,
    argument_two,
    argument_three,
    argument_four,
    argument_five,
)

8.3 — 空行

  • 2 個空行在頂層函式和類別的前後。
  • 1 個空行在類別內的方法之間。
  • 在函式內部少量使用空行來分隔邏輯區段。

8.4 — 匯入順序

按照以下順序分組匯入，每組之間留一個空行：

  1. 標準函式庫（os、sys、math）
  2. 第三方套件（numpy、pandas、requests）
  3. 你自己的模組（from my_project import utils）
import os
import sys

import numpy as np
import requests

from my_project import helpers

8.5 — 運算子周圍的空格

# 好
x = 5
y = x + 3
result = (a + b) * (c - d)

# 不好 — 太擁擠
x=5
y=x+3
result=(a+b)*(c-d)
讓工具幫助你

安裝像 Ruff 或 Black 這樣的 linter/格式化工具。它們會自動格式化你的程式碼以遵循 PEP 8。 在 VS Code 中，安裝 "Ruff" 擴充功能並啟用「儲存時格式化」。 你再也不需要擔心格式問題了。

9. 常見反模式

反模式是能運作但之後會造成問題的程式碼習慣。 從一開始就學會辨識和避免它們。

9.1 — 可變的預設引數

這是 Python 中最臭名昭著的陷阱之一。預設的可變 引數（如串列或字典）在所有呼叫之間共享。 

# 錯誤 — 預設串列在呼叫之間共享！
def add_item(item, items=[]):
    items.append(item)
    return items

print(add_item("a"))  # ['a']       — 看起來沒問題
print(add_item("b"))  # ['a', 'b']  — 等等，'a' 是從哪來的？！
# 正確 — 使用 None 作為預設值，在函式內建立新串列
def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items

print(add_item("a"))  # ['a']
print(add_item("b"))  # ['b']  — 正確！
這個錯誤很難發現

函式第一次呼叫時運作正確。錯誤只在 後續呼叫時才會出現。串列、字典或集合參數 的預設值永遠使用 None。

9.2 — 裸 except

裸 except 會捕獲所有例外，包括 鍵盤中斷和系統退出。這會隱藏錯誤。 

# 錯誤 — 捕獲所有東西，隱藏真正的問題
try:
    result = int(input("Enter a number: "))
except:
    print("Something went wrong")  # 什麼出了問題？誰知道！
# 正確 — 捕獲你預期的特定例外
try:
    result = int(input("Enter a number: "))
except ValueError:
    print("That's not a valid number. Please try again.")

9.3 — 使用全域變數

全域變數使程式碼不可預測，因為任何函式都可以 修改它們。改為透過函式參數傳遞資料。 

# 錯誤 — 依賴全域變數
counter = 0

def increment():
    global counter
    counter += 1

increment()
increment()
print(counter)  # 2 — 但是誰改了它？難以追蹤。
# 正確 — 明確地傳遞和回傳值
def increment(counter):
    return counter + 1

counter = 0
counter = increment(counter)
counter = increment(counter)
print(counter)  # 2 — 資料流清晰

9.4 — 不使用 with 開啟檔案

不使用 with 開啟檔案，如果發生錯誤 檔案可能保持開啟狀態，消耗系統資源。 

# 錯誤 — 如果 close() 之前發生錯誤，檔案可能保持開啟
f = open("data.txt")
data = f.read()
f.close()  # 如果上面那行發生錯誤怎麼辦？
# 正確 — 'with' 保證檔案被關閉，即使發生錯誤
with open("data.txt") as f:
    data = f.read()
# 檔案在此自動關閉

9.5 — 硬編碼路徑

硬編碼路徑在你的程式碼在不同電腦或 作業系統上運行時會出問題。 

# 錯誤 — 只在你的電腦上有效
data = open("C:/Users/alice/projects/data.txt")

# 正確 — 使用相對路徑或 pathlib
from pathlib import Path

data_file = Path(__file__).parent / "data" / "data.txt"
with open(data_file) as f:
    data = f.read()

9.6 — 複製貼上程式碼

如果你複製貼上一段程式碼只改幾個值，你應該 提取一個函式。每個副本都是一個未來等待發生的錯誤 — 當你修正一個副本時，你會忘記修正其他的。 

# 錯誤 — 三份幾乎相同的邏輯
print(f"Student: Alice, Grade: {85}, Status: {'Pass' if 85 >= 60 else 'Fail'}")
print(f"Student: Bob, Grade: {55}, Status: {'Pass' if 55 >= 60 else 'Fail'}")
print(f"Student: Carol, Grade: {72}, Status: {'Pass' if 72 >= 60 else 'Fail'}")
# 正確 — 使用函式和迴圈
def print_student_result(name, grade, passing=60):
    status = "Pass" if grade >= passing else "Fail"
    print(f"Student: {name}, Grade: {grade}, Status: {status}")

students = [("Alice", 85), ("Bob", 55), ("Carol", 72)]
for name, grade in students:
    print_student_result(name, grade)
練習 — 找出反模式

你能在這段程式碼中找到幾個反模式？

import os
data = []
def load():
    global data
    f = open("C:/Users/bob/data.csv")
    try:
        data = f.readlines()
    except:
        print("error")

def process(items=[]):
    for d in data:
        items.append(d.strip())
    return items
顯示答案

有 5 個反模式：

  1. 全域變數 — global data 修改模組層級的狀態。
  2. 沒有 with 語句 — 檔案在沒有 with 的情況下開啟，且從未關閉。
  3. 硬編碼路徑 — "C:/Users/bob/data.csv" 在其他電腦上不能用。
  4. 裸 except — 捕獲所有例外而不指定哪些。
  5. 可變的預設引數 — items=[] 在呼叫之間共享。
