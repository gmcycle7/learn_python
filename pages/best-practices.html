Section I — Python Best Practices

Write clean, readable, and maintainable Python code from the start. These guidelines will save you hours of debugging and make your code a pleasure to revisit.

1. Naming Conventions

Good names make code self-documenting. Python has strong conventions defined in PEP 8.

WhatConventionExample
Variablessnake_casetotal_price, user_name
Functionssnake_casecalculate_area(), get_user_input()
ClassesPascalCaseDataProcessor, HttpClient
ConstantsUPPER_SNAKE_CASEMAX_RETRIES, PI
Modules/filessnake_casedata_utils.py, file_handler.py

Good vs Bad Names

# Bad — vague, single-letter, or misleading
x = 3.14
l = [1, 2, 3]          # 'l' looks like '1' in many fonts
def calc(a, b):
    return a * b

# Good — clear, descriptive, intention-revealing
pi = 3.14
scores = [1, 2, 3]
def calculate_rectangle_area(width, height):
    return width * height
When single letters are OK

Loop counters (i, j, k), coordinates (x, y, z), and short math formulas where the variable meaning is obvious from context are acceptable. Everywhere else, use descriptive names.

MATLAB comparison

MATLAB culture tends toward short variable names (A, b, idx). Python culture strongly prefers longer, descriptive names. temperature_readings is better than T or temps.

2. Code Structure

A well-organized Python file follows a predictable layout. This makes it easy for anyone (including future you) to find things quickly. 

#!/usr/bin/env python3
"""Module docstring — one line describing what this file does."""

# === Standard library imports ===
import os
import sys
from pathlib import Path

# === Third-party imports ===
import numpy as np
import pandas as pd

# === Local imports ===
from my_package import helpers

# === Constants ===
MAX_RETRIES = 3
DEFAULT_TIMEOUT = 30

# === Functions / Classes ===
def process_data(filepath):
    """Read data from filepath and return processed results."""
    with open(filepath) as f:
        data = f.read()
    return data.strip()

def main():
    """Entry point for the script."""
    result = process_data("data.txt")
    print(result)

# === Main guard ===
if __name__ == "__main__":
    main()

The __name__ Guard Explained

The if __name__ == "__main__": block runs only when you execute the file directly (python my_script.py). It does not run when the file is imported as a module. This lets you reuse functions from the file without triggering the main code. 

# Without the guard — this runs even when imported!
result = process_data("data.txt")   # runs on import

# With the guard — only runs when executed directly
if __name__ == "__main__":
    result = process_data("data.txt")   # safe
MATLAB comparison

In MATLAB, each .m file is either a script or a function. Python files can be both — the __name__ guard is how you control which behavior you get.

3. Comments & Docstrings

Comments explain why, not what. The code itself should be readable enough to show what it does.

Bad Comments (state the obvious)

x = x + 1  # increment x by 1

# loop through the list
for item in my_list:
    print(item)  # print the item

Good Comments (explain why)

# Add 1 because the sensor reports 0-based channel numbers,
# but the display expects 1-based numbering
channel_display = channel_raw + 1

# Skip the first row — it contains headers, not data
for row in rows[1:]:
    process(row)

Docstrings

Every function should have a docstring — a string on the first line inside the function that describes what it does. 

def celsius_to_fahrenheit(celsius):
    """Convert a temperature from Celsius to Fahrenheit.

    Args:
        celsius (float): Temperature in degrees Celsius.

    Returns:
        float: Temperature in degrees Fahrenheit.
    """
    return celsius * 9 / 5 + 32
Why docstrings matter

Python's built-in help() function reads docstrings. Typing help(celsius_to_fahrenheit) in the REPL will display your docstring. IDEs like VS Code also show docstrings as tooltips when you hover over a function call.

Exercise — Write a Docstring

4. When to Write Functions

Functions are the fundamental unit of code reuse. Follow the DRY principle (Don't Repeat Yourself).

Write a function when:

  • You use the same code 2+ times. If you copy-paste, you should extract a function instead.
  • A block of code does one clear thing. Give it a name that describes that thing.
  • You want to test a piece of logic independently. Functions are easy to test; tangled scripts are not.
  • The code is getting hard to follow. Breaking it into named functions makes the flow readable.

Before: repetitive code

# Calculating area in three different places
area1 = 3.14159 * radius1 ** 2
print(f"Area 1: {area1:.2f}")

area2 = 3.14159 * radius2 ** 2
print(f"Area 2: {area2:.2f}")

area3 = 3.14159 * radius3 ** 2
print(f"Area 3: {area3:.2f}")

After: a function

import math

def circle_area(radius):
    """Return the area of a circle with the given radius."""
    return math.pi * radius ** 2

for r in [radius1, radius2, radius3]:
    print(f"Area: {circle_area(r):.2f}")
Keep functions short

A good rule of thumb: if a function does not fit on your screen (~20–30 lines), it is probably doing too much. Break it into smaller helper functions.

5. When to Split Into Files

As your project grows, a single .py file becomes hard to navigate. Here is when and how to split.

Signs you should split:

  • Your file is longer than ~200–300 lines.
  • You have groups of functions that serve different purposes (e.g., file I/O functions vs math functions vs display functions).
  • You want to reuse some functions in other projects.

Example: splitting a project

# Before: one big file (main.py, 400+ lines)
# After: organized into modules

my_project/
    main.py              # Entry point — calls functions from other files
    data_loader.py       # Functions for reading files and parsing data
    calculations.py      # Math and processing functions
    display.py           # Printing and formatting functions
    constants.py         # Shared constants (MAX_SIZE, FILE_PATH, etc.)

How to import from your own files:

# main.py
from data_loader import read_csv, parse_header
from calculations import compute_average
from display import print_report
from constants import MAX_SIZE

data = read_csv("measurements.csv")
avg = compute_average(data)
print_report(avg)
What is a package?

A package is a folder containing an __init__.py file (can be empty). It lets you organize modules into subdirectories: from my_package.utils import helper. You do not need packages for small projects — just separate .py files in the same folder work fine.

6. Debugging Tips

Every programmer spends time debugging. These techniques will help you find and fix bugs faster.

6.1 — Print Debugging

The simplest and most common technique. Add print() statements to see what your variables contain at each step. 

def process_scores(scores):
    total = 0
    for i, score in enumerate(scores):
        print(f"DEBUG: i={i}, score={score}, total={total}")  # temporary
        total += score
    return total / len(scores)
Remember to remove debug prints

Always search for "DEBUG" or "TODO" before sharing your code. A quick Ctrl+F for "DEBUG" in your editor catches leftover print statements.

6.2 — Using assert

assert statements verify that your assumptions are correct. If the condition is False, Python raises an AssertionError. 

def calculate_average(numbers):
    assert len(numbers) > 0, "Cannot average an empty list"
    return sum(numbers) / len(numbers)

# This will work fine:
calculate_average([10, 20, 30])

# This will raise AssertionError with your message:
calculate_average([])  # AssertionError: Cannot average an empty list

6.3 — Using breakpoint()

Python 3.7+ has a built-in breakpoint() function that drops you into an interactive debugger (pdb) right at that line. 

def mystery_bug(data):
    result = []
    for item in data:
        breakpoint()  # Execution pauses here — you can inspect variables
        result.append(item * 2)
    return result

When execution pauses, you can type variable names to see their values, n to go to the next line, c to continue, and q to quit the debugger.

6.4 — Read the Error Message Carefully

Python error messages are unusually helpful. They tell you:

  • The type of error (TypeError, ValueError, NameError, etc.)
  • The exact line number where it occurred
  • A description of what went wrong
Read the last line first

The most important information is at the bottom of the error message. Start there, then work your way up to find the line in your code (not library code) that caused the error.

7. How to Read Traceback Errors

A traceback is the full stack of function calls that led to an error. Learning to read tracebacks is one of the most valuable debugging skills you can develop.

Example: a real traceback

# The code
def get_average(numbers):
    return sum(numbers) / len(numbers)

def process_file(filename):
    with open(filename) as f:
        values = [int(line) for line in f]
    return get_average(values)

result = process_file("data.txt")

If data.txt contains a non-numeric line, you get:

Traceback (most recent call last):           # 1. Header
  File "main.py", line 9, in <module>        # 2. Top of the call stack
    result = process_file("data.txt")
  File "main.py", line 6, in process_file    # 3. Deeper into the stack
    values = [int(line) for line in f]
ValueError: invalid literal for int()        # 4. The actual error
  with base 10: 'hello\n'

How to read it (bottom-up):

  1. Line 4 (bottom) — The error type and message: ValueError tells you a value was wrong. The message says it tried to convert 'hello\n' to an integer and failed.
  2. Line 3 — Where the error happened: File main.py, line 6, inside the function process_file. The culprit is the list comprehension [int(line) for line in f].
  3. Line 2 — What called it: Line 9 called process_file("data.txt").
  4. Line 1 — Just the header: "Traceback (most recent call last)" — always the same.
The fix

Add error handling or data validation. For example, wrap int(line) in a try/except or filter out non-numeric lines before converting.

def get_initials(names):

people = ["Alice", "Bob", "", "Diana"]

for name in names:

        initials.append(name[0].upper())

8. Beginner-Friendly Style Guide (PEP 8)

PEP 8 is Python's official style guide. Here are the rules that matter most for beginners.

8.1 — Use 4 spaces for indentation

# Correct — 4 spaces
def greet(name):
    print(f"Hello, {name}")

# Wrong — 2 spaces (common in JavaScript, not in Python)
def greet(name):
  print(f"Hello, {name}")
Never mix tabs and spaces

Configure your editor to insert 4 spaces when you press Tab. Mixing tabs and spaces causes IndentationError or, worse, invisible bugs. In VS Code, this is the default for Python files.

8.2 — Line length: aim for 79–99 characters

PEP 8 recommends 79 characters. Many teams allow up to 99 or 120. The key is: if you have to scroll sideways, the line is too long. 

# Too long — hard to read
result = some_function(argument_one, argument_two, argument_three, argument_four, argument_five)

# Better — break it up
result = some_function(
    argument_one,
    argument_two,
    argument_three,
    argument_four,
    argument_five,
)

8.3 — Blank lines

  • 2 blank lines before and after top-level functions and classes.
  • 1 blank line between methods inside a class.
  • Use blank lines sparingly inside functions to separate logical sections.

8.4 — Import order

Group imports in this order, with a blank line between each group:

  1. Standard library (os, sys, math)
  2. Third-party packages (numpy, pandas, requests)
  3. Your own modules (from my_project import utils)
import os
import sys

import numpy as np
import requests

from my_project import helpers

8.5 — Spaces around operators

# Good
x = 5
y = x + 3
result = (a + b) * (c - d)

# Bad — too cramped
x=5
y=x+3
result=(a+b)*(c-d)
Let your tools help

Install a linter/formatter like Ruff or Black. They auto-format your code to follow PEP 8. In VS Code, install the "Ruff" extension and enable "Format On Save." You will never have to think about formatting again.

9. Common Anti-Patterns

Anti-patterns are code habits that work but cause problems later. Learn to recognize and avoid them from the start.

9.1 — Mutable Default Arguments

This is one of the most infamous Python gotchas. Default mutable arguments (like lists or dicts) are shared across all calls. 

# WRONG — the default list is shared between calls!
def add_item(item, items=[]):
    items.append(item)
    return items

print(add_item("a"))  # ['a']       — looks OK
print(add_item("b"))  # ['a', 'b']  — wait, where did 'a' come from?!
# CORRECT — use None as the default, create a new list inside
def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items

print(add_item("a"))  # ['a']
print(add_item("b"))  # ['b']  — correct!
This bug is hard to spot

The function works correctly the first time. The bug only appears on subsequent calls. Always use None as the default for list, dict, or set parameters.

9.2 — Bare except

A bare except catches every exception, including keyboard interrupts and system exits. This hides bugs. 

# WRONG — catches everything, hides the real problem
try:
    result = int(input("Enter a number: "))
except:
    print("Something went wrong")  # What went wrong? Who knows!
# CORRECT — catch the specific exception you expect
try:
    result = int(input("Enter a number: "))
except ValueError:
    print("That's not a valid number. Please try again.")

9.3 — Using Global Variables

Global variables make code unpredictable because any function can change them. Pass data through function parameters instead. 

# WRONG — relying on a global variable
counter = 0

def increment():
    global counter
    counter += 1

increment()
increment()
print(counter)  # 2 — but who changed it? Hard to track.
# CORRECT — pass and return values explicitly
def increment(counter):
    return counter + 1

counter = 0
counter = increment(counter)
counter = increment(counter)
print(counter)  # 2 — clear data flow

9.4 — Not Using with for Files

Opening files without with can leave files open if an error occurs, consuming system resources. 

# WRONG — file may stay open if an error occurs before close()
f = open("data.txt")
data = f.read()
f.close()  # What if an error happens on the line above?
# CORRECT — 'with' guarantees the file is closed, even on error
with open("data.txt") as f:
    data = f.read()
# File is automatically closed here

9.5 — Hardcoding Paths

Hardcoded paths break when your code runs on a different computer or operating system. 

# WRONG — works only on your machine
data = open("C:/Users/alice/projects/data.txt")

# CORRECT — use relative paths or pathlib
from pathlib import Path

data_file = Path(__file__).parent / "data" / "data.txt"
with open(data_file) as f:
    data = f.read()

9.6 — Copy-Pasting Code

If you copy-paste a block of code and change a few values, you should extract a function instead. Every copy is a future bug waiting to happen — when you fix one copy, you will forget to fix the others. 

# WRONG — three copies of nearly identical logic
print(f"Student: Alice, Grade: {85}, Status: {'Pass' if 85 >= 60 else 'Fail'}")
print(f"Student: Bob, Grade: {55}, Status: {'Pass' if 55 >= 60 else 'Fail'}")
print(f"Student: Carol, Grade: {72}, Status: {'Pass' if 72 >= 60 else 'Fail'}")
# CORRECT — use a function and a loop
def print_student_result(name, grade, passing=60):
    status = "Pass" if grade >= passing else "Fail"
    print(f"Student: {name}, Grade: {grade}, Status: {status}")

students = [("Alice", 85), ("Bob", 55), ("Carol", 72)]
for name, grade in students:
    print_student_result(name, grade)
Exercise — Spot the Anti-Patterns

How many anti-patterns can you find in this code?

import os
data = []
def load():
    global data
    f = open("C:/Users/bob/data.csv")
    try:
        data = f.readlines()
    except:
        print("error")

def process(items=[]):
    for d in data:
        items.append(d.strip())
    return items
Show Answer

There are 5 anti-patterns:

  1. Global variable — global data modifies module-level state.
  2. No with statement — file opened without with and never closed.
  3. Hardcoded path — "C:/Users/bob/data.csv" will not work on other machines.
  4. Bare except — catches all exceptions without specifying which ones.
  5. Mutable default argument — items=[] is shared across calls.
